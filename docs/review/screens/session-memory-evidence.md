# Session Memory Evidence — Future Feature

**Status:** PLANNED (not yet implemented)
**Priority:** Medium
**Depends on:** Pattern detection fix (complete), daily digest storage (not built)

## Concept

When a trader shows the same behavioral pattern as a past bad session, the AI uses that historical session as concrete evidence:

> "Last time you did this (Feb 28), you had 4 consecutive losses and ended the day down 73%. You're showing the same pattern now — consider stopping."

This turns abstract warnings ("you're revenge trading") into vivid, personal stories backed by the trader's own data.

## Requirements

### 1. Session Summarization
- Daily digest: after market close, summarize the day's patterns, P&L trajectory, and emotional state
- Store in `session_summaries` table: `date`, `broker_account_id`, `patterns_detected[]`, `peak_pnl`, `final_pnl`, `trade_count`, `narrative`
- Narrative generated by AI (Claude Haiku) — 2-3 sentences

### 2. Pattern Matching Across Sessions
- When a live pattern is detected (e.g., tilt spiral), query `session_summaries` for past days with the same pattern
- Rank by severity of outcome (biggest drawdown, most trades, worst P&L give-back)
- Return top 1-2 historical matches

### 3. AI Prompt Enrichment
- Inject historical session context into coach prompts:
  ```
  HISTORICAL CONTEXT: On {date}, you showed the same {pattern_name} pattern.
  That session ended with {final_pnl} after peaking at {peak_pnl}.
  You made {trade_count} trades that day.
  ```
- AI uses this to craft personalized, evidence-backed warnings

## Data Flow

```
Market close → Celery task → Summarize day → Store session_summary
                                                     ↓
Live pattern detected → Query similar past sessions → Enrich AI prompt
                                                     ↓
                                              Coach/WhatsApp message
                                              with historical evidence
```

## Migration (future)

```sql
CREATE TABLE session_summaries (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    broker_account_id UUID REFERENCES broker_accounts(id) ON DELETE CASCADE,
    session_date DATE NOT NULL,
    patterns_detected TEXT[],
    peak_pnl NUMERIC(15,4),
    final_pnl NUMERIC(15,4),
    trade_count INTEGER,
    max_consecutive_losses INTEGER,
    narrative TEXT,
    created_at TIMESTAMPTZ DEFAULT now(),
    UNIQUE(broker_account_id, session_date)
);
CREATE INDEX idx_session_summaries_patterns ON session_summaries USING GIN(patterns_detected);
```

## Why This Matters

- Traders respond to their own history more than generic advice
- Creates a "this is what happened last time" feedback loop
- Builds long-term engagement (the app gets smarter over time)
- Addresses the cold-start problem partially — after 2 weeks, the app has enough sessions to reference
